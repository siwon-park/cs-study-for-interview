# 00_이메일 프로토콜

> SMTP, IMAP, POP3

이메일 송신, 수신과 관련된 프로토콜 정리

## 1. SMTP

> Simple Mail Transfer Protocol

전자 우편을 송신하는데 사용되는 TCP/IP 표준 프로토콜.

클라이언트가 자기 메일 서버에 메일을 제출할 때 사용하는 프로토콜이다.

PUSH 방식 (보내는 쪽이 능동적)이며, 메일 서버끼리 메일을 전달(relay)할 때도 사용한다.

### 1) 특징

- 텍스트 기반 명령어(`HELO`, `MAIL FROM`, `RCPT TO`, `DATA`, `QUIT`)
- TCP/IP 기반 프로토콜
  - TCP/IP의 특징처럼 연결 확인 - 연결 - 신호 전달과 같은 단계를 거친다.
  - SMTP 서버로의 접근 가능성 확인 - 서버와 연결 설정 - 메일 전송
- SMTP 서버와 클라이언트로 구성된다.
- 사용 포트는
  - 25: smtp 서버 to 서버
  - 587: 클라이언트 to 서버
  - 465: 클라이언트 to 서버(SMTPS 보안 통신)


<br>

### 2) SMTP 서버와 클라이언트 및 동작 과정

#### (1) SMTP 클라이언트

메일을 보낸 주체

※ 메일을 받는 상대방쪽은 클라이언트가 아니다.

- 왜냐하면 결국 SMTP 클라이언트가 SMTP 서버로 데이터를 보내고 받는 쪽은 해당 서버로부터 메세지를 수신만 받기 때문이다.
- 즉, 클라이언트 - 보내는 서버 - 받는 서버라고 이해하면 편하다.

![image](https://github.com/siwon-park/cs-study-for-interview/assets/93081720/50af493e-1eae-4e6c-85c8-6ddc0654069c)

#### (2) SMTP 서버

메일 데이터를 송/수신하는 서버

- 예를 들어, gmail의 SMTP 서버는 smtp.gmail.com이고, naver의 SMTP 서버는 smtp.naver.com이다.

클라이언트가 본인의 메일 주소로 메일을 보내면 해당 메일 데이터는 클라이언트가 사용하는 메일 도메인의 메일 서버로 1차적으로 보내진 다음에, 수신측의 메일 서버로 보내지는 것이다.

- 예) gmail → naver로 메일 송신 시; 클라이언트 메일 작성(gmail) → gmail SMTP서버 → naver SMTP서버

<br>

#### (3) 동작 과정

송신자(클라이언트)가 메일을 작성해서 보내면 해당 메일이 SMTP 서버(보내는 메일 서버, Outgoing Mail Server)로 전송되며, 메일을 받은 SMTP 서버는 SENDMAIL 프로그램을 구동하여 해당 메일을 알맞은 수신자에게 보내는 것이다.

※ DNS와 MX 레코드

- MX 레코드란 메일이 수신될 위치를 결정하는 레코드를 말한다.

(동작 예시)

1. mydomain.kr의 MX 레코드를 네이버웍스의 MX 레코드 값으로 설정
2. 외부에서 id@mydomain.kr로 메일을 보내는 경우
3. DNS에 네이버웍스의 MX 레코드가 조회되므로 네이버웍스 서버로 메일을 전송
4. 네이버웍스의 메일함에서 수신된 메일 확인

## 2. POP3

> Post Office Protocol v3

메일을 수신하기 위한 프로토콜.

단, POP3의 수신은 "받아오기", "꺼내오기"를 의미한다.

### 1) 특징

- 서버로부터 클라이언트로 메일을 다운로드하고 삭제한다. (중요)
- PULL 방식이다.
  - 클라이언트가 서버로 메일을 달라고 요청하는 클라이언트 to 서버 요청이다.
- 하나의 기기에서 (오프라인으로) 메일을 본다는 가정을 한다.
  - 따라서 동기화가 되지 않으며, 다른 기기에서보면 메일이 없을 수 있다.
- 사용 포트는
  - 110: 클라이언트 to 서버 (메일 수신)
  - 995: 클라이언트 to 서버 (메일 수신 보안 통신, POP3S)

<br>

## 3. IMAP

> Internet Message Access Protocol

메일을 수신하기 위한 프로토콜.

단, IMAP의 수신은 "동기화하며 받는 것"을 의미한다.

### 1) 특징

- 메일 데이터는 서버에 그대로 두고, 클라이언트는 해당 메일 데이터를 동기화만 한다.
  - 원본(master copy)는 서버에 저장되어 있고, 클라이언트는 메일 데이터를 보여주기 위한 로컬 캐시만 가져간다.
    - 모든 변경의 진실의 원천(source of truth)은 서버
  - POP3가 메일 데이터를 클라이언트로 가져와서 저장한 다음 서버에서 삭제하는 것과는 다른 차이점.
  - 단, 클라이언트에서 삭제를 하면 서버에서도 삭제된다. (COPY, MOVE 등 폴더 간 이동 및 복사도 서버로 전달되어 처리됨)
    - `\ Deleted` (삭제 예정) → `EXPUNGE` (완전 삭제)
- 따라서 여러 기기에서도 같은 상태로 메일을 열람할 수 있다.
- POP3와 달리 폴더의 개념이 존재하며, 읽음/안읽음 및 플래그 등의 상태관리가 가능하다.
- 헤더만 먼저 받고 본문은 필요할 때 가져오는 식의 부분 다운로드 기능도 지원한다.
- POP3에 비해 모던한 방식의 프로토콜이며, 현대의 대부분의 메일 서비스는 IMAP을 기본으로 한다.
- 사용 포트는
  - 143: 클라이언트 to 서버 (메일 수신)
  - 993: 클라이언트 to 서버 (메일 수신, IMAPS)
